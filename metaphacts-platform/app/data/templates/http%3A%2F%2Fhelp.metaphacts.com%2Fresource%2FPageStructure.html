Page Structure

When building applications, one major part is the composition of the user interface. To support the application developer in this task the platform offers a rich toolset with a particular focus on declarative UI building.

The goal of the platform is to provide well-defined out-of-box templates and configurations which allow the user to browse and explore arbitrary knowledge graphs. Going beyond this application developers can use the available UI components to produce domain-specific end-user oriented applications.

Below we describe the overall page structure and provide the details on how the UI can be adjusted to create an end-user application. More specific details on UI components, the available frontend and backend templating features as well as advanced topics can be found in the documentation overview.

Page Elements

All pages in the platform are constructed from a number of pre-defined and unified elements that can be enabled or disabled depending on a use case in mind.

Similar to building blocks in construction industry, every page element has its own assigned place on the page, its own purpose, realm of visibility and dedicated means of configuring it.

The following list summarizes the available page elements:

  • Platform header
  • Breadcrumbs
  • Developer toolbar
  • Knowledge Graph Bar
  • View content

The elements will be described one by one to explain how to control the rendering of a page in the platform.

Platform Header

The platform header is always rendered at the top of any page and exposes the means of navigation to the most frequently used platform tools. Visibility of some elements in the header depends on permissions. For example, the admin menu will be shown only to those users that are in a role granting an access to the Administration area.

You can learn more about permissions and user roles in Security Section.

Breadcrumbs

Breadcrumbs are a visual aid for users to keep track and maintain awareness of their locations within the application. Such breadcrumbs and the hierarchy they represent are very much application-specific.

In the platform we define breadcrumbs either as part of the content block, or, for more advanced applications, by providing type-based definitions using Page View Configuration.

    Developer Toolbar

    The Developer Toolbar is another collection of tools specifically targeted to application developers.

    At the moment this toolbar consists of one item, the Edit Page button, which allows to edit the HTML source of a page.

    The Edit Page button is visible, if the user has "write" permissions to the current resource.

    . To learn how to set permissions please refer to Page View Configuration.

    Knowledge Graph Bar

    The Knowledge Graph Bar is a central user interface component that shows relevant information about the current resource. It is targeted towards knowledge graph expert users as well as application developer users to provide consistent access to all resource information. Typically, the Knowledge Graph Bar is not shown in end-user applications, however, expert users with the required permissions can always toggle it on and off for themselves.

    The default Knowledge Graph Bar consists of the following elements:

    • Resource label;
    • Resource thumbnail;
    • Resource type;
    • Resource URI;
    • Toggle buttons for switching between the 3 pre-defined resourse views;
    • Collapse / expand chevron to control the show / hide state of the Knowledge Graph Bar.

    Knowledge Graph Bar Visibility

    For a set of the following out-of-box product sections the Knowledge Graph Bar is shown to all users:

    • Named Graphs
    • Vocabularies (SKOS vocabularies and SKOS collections as well as their elements)
    • Ontologies and all their elements (classes, ontology properties, etc.)

    The platform is configured to not show the Knowledge Graph Bar:

    • on static pages (e.g. admin and help pages);
    • by default for user-defined application pages;
    • on applied templates (with the exception of the default rdfs:Resource template)

    However, as mentioned above, the Knowledge Graph Bar can explicitly be brought into view for all RDF resources by expert users with the respective permissions. For the latter user group the UI offers a button (a blue collapse / expand chevron) in the top-center below the application header to blend in the knowledge graph bar. Details on the permissions can be found in the security documentation (permission ui:page:knowledge-graph-bar:toggle in the UI permissions section).

    Thus, a clear distinction is now being drawn between the platform as a product for managing knowledge graphs and the end-user space to be later defined by the application developer.

    The following matrix gives a comprehensive overview about Knowledge Graph Bar visibility:

    Resource / Page Description

    Out-of-box product sections (ontologies, vocabularies, etc.) The Knowled Graph Bar is shown.
    Static product pages (admininistration or help pages) The Knowled Graph Bar is NOT shown.
    RDF resource with application page or template

    Resource Views

    In case the user has permissions to use the Knowledge Graph Bar, the toggle buttons located on the right side of the Knowledge Graph Bar allow the user to switch between different resource views. Directly below the available views are described in detail.

    These simple rules explain which view is shown by default:

    • the Graph View is the default option for any type of resource, i.e. it is shown for the RDF resources, for which no dedicated template or application page exists
    • the HTML View is shown, if the application defines a corresponding template or page for this particular resource type, for example, if a developer has defined a special template page for foaf:Person.

    A specific application can fine-tune the views for selected resource types via the Page View Configuration, but the out-of-the-box defaults must suffice in 90% of cases.

    Graph View

    The Graph View offers an interactive graph visualization to navigate and explore the knowledge graph, starting with the resource in focus.

    The graph view is shown for RDF resources by default, if there is no specific template or application page defined.

    Note: whether the diagram can be saved or not depends on the current user's permissions.

    Statements View

    The Statements View provides a tabular visualization of the outgoing and incoming statements of the current resource. It is a means for advanced users to display the raw relations of the current resource, as defined in the database. All entries are clickable and can be used for navigation to the linked resources.

    HTML / Page View

    The HTML or Page View shows the domain specific content (which in other occurancies can be named content block) as defined and designed by the application developer. It can make use of the rich set of UI components as well as templating features. See here for details on templates and application pages.

    Note that if a page does not yet exist, the platform shows a corresponding message indicating that there is no content defined. Once the application developer provides such content, the rendered sources are presented to the user.
